# SEO specification — hulubul.com

**Date:** 2026-05-14
**Status:** Spec only. Implementation plan ships separately.
**Scope:** Discoverability and shareability of the marketing surface
(`/`, `/sondaj/expeditori`, `/rute`, the four editorial pages, future
`/sondaj/transportatori`).

---

## 1. Goals

1. Be indexed correctly by Google and Bing within one week of public launch.
2. Generate clean, branded social previews (Open Graph + Twitter Card) when
   anyone pastes a hulubul.com URL into WhatsApp, Telegram, Facebook,
   Slack, LinkedIn or X.
3. Surface in branded search ("hulubul") with the right knowledge-panel
   shape: site name, logo, tagline.
4. Pass Core Web Vitals on every public route (LCP < 2.5s, INP < 200ms,
   CLS < 0.1) on the median 4G connection.
5. Stay accessible (WCAG 2.1 AA on text/colour/contrast) — accessibility
   is the largest correlated SEO signal nobody talks about.

Out of scope for v1: i18n alternates (page is `ro`-only), AMP, Google News.

## 2. Metadata model

### 2.1 Single source of truth

All page-level metadata flows through Next.js `generateMetadata` (App
Router native API). No third-party SEO libraries (`next-seo`,
`next-sitemap`) — App Router covers their feature surface natively and
avoids config drift.

### 2.2 Root defaults (`app/layout.tsx`)

The root `metadata` export sets defaults inherited by every route:

| Field | Value |
|---|---|
| `metadataBase` | `new URL("https://hulubul.com")` — required so all OG image URLs resolve absolute |
| `title.default` | `"hulubul.com"` |
| `title.template` | `"%s — hulubul.com"` |
| `description` | Fallback site description (RO) |
| `applicationName` | `"hulubul.com"` |
| `authors` | `[{ name: "hulubul.com" }]` |
| `creator`, `publisher` | `"hulubul.com"` |
| `formatDetection` | `{ telephone: false, email: false, address: false }` — disable iOS link auto-detection |
| `openGraph.type` | `"website"` |
| `openGraph.siteName` | `"hulubul.com"` |
| `openGraph.locale` | `"ro_RO"` |
| `openGraph.images` | `[{ url: "/og-default.png", width: 1200, height: 630 }]` |
| `twitter.card` | `"summary_large_image"` |
| `twitter.creator` | `"@hulubul_com"` (when account exists) |
| `robots` | `{ index: true, follow: true, googleBot: { "max-image-preview": "large", "max-snippet": -1, "max-video-preview": -1 } }` |
| `icons` | `{ icon: "/favicon.ico", apple: "/apple-touch-icon.png" }` |
| `manifest` | `"/site.webmanifest"` (PWA-lite) |

### 2.3 Per-page overrides

| Route | Metadata source |
|---|---|
| `/` | `landing-page.seo.{metaTitle, metaDescription, shareImage}` from CMS (already wired) |
| `/sondaj/expeditori` | Static `generateMetadata` in the page file. `robots: { index: false }` — survey is a transient form, not a search target. |
| `/sondaj/transportatori` (future) | Same as above. `noindex`. |
| `/rute` | Static title `"Rute de transport"` + dynamic description listing top corridors. `index: true`. |
| `/admin/rute` | `robots: { index: false, follow: false }` — admin surface, never indexed. |
| `/confidentialitate` | From `EditorialPage` (CMS or fallback). `index: true`. |
| `/termeni` | Same. `index: true`. |
| `/despre-proiect` | Same. `index: true`. |
| `/pentru-transportatori` | Same. `index: true`. |

### 2.4 Canonical URLs

Every public page sets `alternates.canonical` to its absolute URL. This
prevents:
- Duplicate-content penalties when the page is reachable via multiple
  paths (e.g. `/?role=expeditor` query-param variants of `/`).
- Indexing of staging/preview URLs if `metadataBase` is mistakenly set.

The home page canonicalises to `https://hulubul.com/` regardless of
incoming `?role=` / `#signup` decoration.

## 3. Open Graph + Twitter Cards

### 3.1 Image strategy

- **Default OG image** at `public/og-default.png` (1200×630, ≤ 200 KB,
  PNG with text-as-vector to render crisply on retina previews).
- **Per-page OG image** when the CMS provides one
  (`landing-page.seo.shareImage`, future `EditorialPage.ogImage`).
- One **dynamic OG image generator** at `app/og/route.tsx` using
  `next/og` (`@vercel/og` under the hood, native to Next.js). Renders
  `https://hulubul.com/og?title=…&subtitle=…` into a 1200×630 PNG using
  the project fonts (Fraunces + Inter). Used by editorial pages without
  a CMS image.

### 3.2 Required OG fields per page

```
og:title          ≤ 60 chars
og:description    ≤ 155 chars
og:url            absolute, canonical
og:type           website (most pages) | article (future blog)
og:image          1200×630, absolute URL
og:image:alt      RO description of the image content (a11y + sharing UX)
og:locale         ro_RO
og:site_name      hulubul.com
```

Twitter mirrors these via `twitter.{card, title, description, images}`.

## 4. Structured data (JSON-LD)

Schema.org structured data is rendered server-side as a
`<script type="application/ld+json">` element in the relevant page's
`<head>`. Use Next.js's standard pattern (a React `<script>` element
with the JSON serialised into its content via a safe stringifier).

Use **`schema-dts`** (~10 KB types-only, zero runtime) for type-safe
JSON-LD construction.

### 4.1 Schemas per page

| Schema | Where | Purpose |
|---|---|---|
| `Organization` (with `parentOrganization → Meaningfy`) | Root layout (every page) | Knowledge panel, logo, parent attribution, `sameAs` to social |
| `WebSite` (`SearchAction` later) | Root layout | Site identity; SERP sitelinks-search box reserved for when an in-site search exists |
| `BreadcrumbList` | Editorial pages (`/confidentialitate`, `/termeni`, `/despre-proiect`, `/pentru-transportatori`) | Breadcrumb in SERPs |
| `FAQPage` (driven by CMS) | `/` | Rich-result FAQ accordion in Google SERPs |
| `Service` — *trusted-carrier directory for cross-border parcels* (§4.4) | `/` | Sender + recipient value prop in structured form |
| `Service` — *peer-to-peer logistics matchmaking* (§4.3) | `/pentru-transportatori` | Transporter-facing value prop in structured form |
| `Place` (later, per route) | `/rute/[slug]` future detail pages | Transit corridor structured data |

The two `Service` schemas are emitted on different pages so each
audience lands on a JSON-LD body that mirrors the page copy. The
`Organization`, `WebSite`, and `Service`(s) on the home page ship as a
single JSON-LD `@graph` array (one document, multiple entities) so
search engines see them as one coherent entity model.

## 5. Sitemap and robots

### 5.1 `app/sitemap.ts`

App-Router native. Generates `https://hulubul.com/sitemap.xml` at build.
Lists:
- `/` (priority 1.0, changefreq weekly)
- `/rute` (priority 0.8, changefreq daily)
- All four editorial pages (priority 0.6, changefreq monthly)

`/sondaj/*` is excluded — those pages are `noindex` and shouldn't be in
the sitemap either.

Last-modified is taken from the corresponding CMS `updatedAt` for
landing/editorial pages, and from build time for static routes.

### 5.2 `app/robots.ts`

App-Router native. Generates `/robots.txt`:

```
User-agent: *
Allow: /
Disallow: /admin/
Disallow: /api/
Disallow: /sondaj/

Sitemap: https://hulubul.com/sitemap.xml
```

`/sondaj/*` is disallowed even though server-rendered noindex would
catch it — defense in depth, and saves crawler budget.

### 5.3 Search Console verification

- **Google Search Console:** verify via DNS TXT record (preferred) or
  meta tag in root layout. DNS verifies the whole property; meta tags
  only verify a single subdomain.
- **Bing Webmaster Tools:** same approach, separate property.

Verification tokens stored as env vars
(`GOOGLE_SITE_VERIFICATION`, `BING_SITE_VERIFICATION`) and emitted into
`metadata.verification.{ google, other.bing }`. Empty env = tag absent.

## 6. Performance and Core Web Vitals

### 6.1 Budgets

Enforce per-page in CI with Lighthouse CI (`@lhci/cli`):

| Metric | Budget |
|---|---|
| Lighthouse Performance | ≥ 90 |
| Lighthouse SEO | ≥ 95 |
| Lighthouse Accessibility | ≥ 95 |
| LCP | ≤ 2.5s on simulated 4G |
| INP | ≤ 200ms |
| CLS | ≤ 0.1 |
| Total page weight | ≤ 350 KB transferred (excluding lazy chunks) |
| Third-party JS weight | ≤ 70 KB blocking — gates the analytics + tag layer (see tracking spec §3.1) |

### 6.2 Implementation notes already covered

- `next/font/google` for Fraunces + Inter — already in place. No FOIT.
- `next/image` for any future hero / OG render — currently no images
  used in components; verify before adding.
- Marketing layout uses Server Components — already in place.
- Leaflet on `/rute` is dynamic-imported with `ssr: false` — already in
  place. Map is below the fold and below the work-in-progress banner.

### 6.3 What we should add

- **Lighthouse CI workflow** in `.github/workflows/lighthouse.yml` that
  runs on PR against a Vercel preview URL.
- **Web Vitals reporter** in `app/layout.tsx` using `next/web-vitals`.
  Report to GA4 directly via `gtag('event', 'cwv', {...})` (gated by
  analytics consent — see tracking spec §4). When the project later
  migrates to GTM (tracking spec §3.1.1) the call shape doesn't change
  because gtag pushes through `dataLayer` under the hood. Lets us
  monitor real-user CWV (RUM data), not just lab scores.

## 7. Accessibility-as-SEO

Google's Search Quality Evaluator Guidelines weight a11y heavily under
"Page Experience". The minimum checks on every PR:

- Lighthouse a11y score ≥ 95.
- `axe-core` automated check via Playwright on the four most-trafficked
  pages (in CI).
- Manual screen-reader smoke per epic (TalkBack on Android, VoiceOver
  on macOS) — checklist in PR template.

Specific items already on the radar:
- Survey checkbox boxes: all clickable area covers the input (PR #9).
- Form errors are announced (`role="alert"`, already in place).
- Skip-link from `<header>` to main content — **not yet implemented**.
- Colour contrast on the work-in-progress banner — verify with axe.

## 8. URL hygiene

- All public pages live under stable, RO-language slugs. No localised
  `/en/` until i18n is in scope.
- No trailing slashes. Configured in `next.config.ts`.
- HTTPS-only via the deployment platform.
- HSTS header (`Strict-Transport-Security: max-age=31536000;
  includeSubDomains; preload`) set at the edge — infra concern,
  documented here for completeness.
- `Permissions-Policy` header restricts geolocation/camera/microphone
  to `self` only — defense in depth, documented.

## 10. Acceptance

The SEO work is done when, on the deployed production site:

1. `curl -s https://hulubul.com/sitemap.xml` returns a valid sitemap.
2. `curl -s https://hulubul.com/robots.txt` returns a valid robots file
   pointing at the sitemap.
3. The home page HTML contains a single `<script type="application/ld+json">`
   block whose `@graph` includes `Organization` (with
   `parentOrganization → Meaningfy`), `WebSite`, `FAQPage`, and the
   sender/recipient `Service` (§4.4). The block validates against
   [validator.schema.org](https://validator.schema.org) with zero
   warnings.
4. The `/pentru-transportatori` HTML contains a JSON-LD block with the
   transporter `Service` (§4.3). Same validator, zero warnings.
5. Pasting `https://hulubul.com/` into the
   [Facebook Sharing Debugger](https://developers.facebook.com/tools/debug/)
   shows the configured title, description, and image with no warnings.
6. Lighthouse on the home page scores ≥ 90 (Performance), ≥ 95
   (Accessibility), ≥ 95 (SEO). Best Practices is not budgeted but
   monitored.
7. All four editorial pages, `/rute`, and `/sondaj/expeditori` carry
   the right `<title>`, `<meta name="description">`, `<link rel="canonical">`,
   and OG/Twitter tags.
8. `/sondaj/expeditori` (and `/sondaj/transportatori` when shipped)
   carries `robots: noindex, nofollow` in its HTML.
9. Google Search Console shows the property verified and the sitemap
   accepted (manual step, post-launch).
